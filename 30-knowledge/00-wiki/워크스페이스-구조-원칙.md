# 워크스페이스-구조-원칙

> **Facets**: PARA · Johnny Decimal · 폴더 구조 · 위키 · AI 협업 · 인지 부담

## 핵심

좋은 워크스페이스 구조는 세 가지 목적을 **동시에** 추구한다:
1. AI와 같이 일할 때 빠르게 맥락 잡기 (활용 중심)
2. 3년 뒤 돌아봐도 자료가 살아있음 (보관 중심)
3. 머릿속 인지 부담 감소 (정리 중심)

이 세 목적은 사실 **같은 답으로 수렴한다**:

> **"폴더는 얕게, 명명·링크·인덱스를 풍부하게"**

거꾸로 — "정교한 폴더 트리"는 세 목적 모두에 안 맞는다. (외울 거 많음 + 3년 뒤 카테고리 미스매치 + AI grep 비효율)

### 5원칙

1. **폴더 깊이 최대 3단계** (기본 2단계)
2. **모든 카테고리 폴더에 README 한 단락** — "여기엔 뭐가 들어가는가"
3. **00-wiki를 살아있게** — raw에서 wiki로 핵심 추출 (SCHEMA.md의 3계층 구조 활용)
4. **명명은 자기설명적** — 파일명만 보고 내용 추측 가능
5. **CLAUDE.md를 진짜 입구로** 유지

### 메타 원칙 — 도구별 역할 분리

5개 동사(정리·관리·보관·운영·활용)는 서로 다른 요구를 가짐. 한 도구로 다 만족시킬 수 없음.

| 동사 | 진짜 요구 | 최적 도구 |
|---|---|---|
| 정리 | 카테고리화, 분류 | 폴더 + 명명 규칙 |
| 관리 | 갱신 용이, 한눈에 보임 | README / 인덱스 페이지 |
| 보관 | 망각해도 다시 찾힘 | 명명 규칙 + 90-archive |
| 운영 | 1~2 클릭 접근, 빠름 | 폴더 1단계 |
| **활용** | **발견, 연결, 통찰** | **링크 + 검색 (폴더 아님)** |

→ 가치 생산의 중심은 **활용**이고, 활용은 폴더가 아닌 **링크·위키**에서 작동함.

### AI 협업이 1순위일 때의 함의

AI는 폴더 트리를 그림으로 보지 않음. AI의 진입 경로:
1. **CLAUDE.md 먼저 읽음**
2. 명명만 보고 grep
3. 폴더 README 있으면 빠르게 이해
4. **00-wiki에서 토픽으로 접근**

→ AI 친화적 워크스페이스의 핵심은 **CLAUDE.md + README + 00-wiki**. 폴더는 보조 도구.

## 적용

- **첫 셋업**: CLAUDE.md의 "내 프로필" 4문항만 채워도 AI가 충분히 협업할 맥락을 잡음.
- **점진 적용**: 폴더·파일을 미리 다 짜지 말고, 첫 활성 프로젝트가 생길 때 그 폴더만 만들어 시작.
- **사전 설계 < 살아있는 인덱스/링크**: 정교한 사전 구조보다 실운영에서 발견되는 마찰을 신호로 점진적으로 다듬는 편이 낫다.

## Open Questions

- raw → wiki 통합의 **운영 루틴**이 구체적으로 어떻게 흐를 것인가? (수동 호출 빈도, supervised vs batch 모드 비율)
- README의 **깊이**는 어디까지 갈 것인가? (1단계 카테고리만? 프로젝트 폴더까지?)
- AI 협업이 1순위지만, 본인이 워크스페이스를 **시각적으로 탐색하는 빈도**는? 그 균형이 폴더 결정에 영향.
- **재평가 트리거**는 무엇인가? 후보: "같은 자료를 두 번 찾았다", "어디 둘지 30초 이상 고민했다", "10-projects가 7개 넘었다" 등.
