# Skill vs Sub-Agent vs Slash Command

> Claude Code의 세 가지 구성요소를 구분하고, 어떤 상황에 무엇을 쓸지 + 커스텀 Sub-Agent를 만드는 5단계.

## 언제 쓰나

- 반복 작업이 보이는데 매번 같은 프롬프트를 치고 있을 때 → **Skill**
- 한 영역에 깊이 있게 일관된 결과를 내는 도구가 필요할 때 → **Sub-Agent**
- 자주 쓰는 명령을 단축어로 만들고 싶을 때 → **Slash Command**

## 세 가지 비교

| 항목 | Slash Command | Skill | Sub-Agent |
|---|---|---|---|
| **본질** | 단축 명령 (`/foo`) | 업무 매뉴얼 (md 파일 폴더) | 전담 전문가 (격리된 세션) |
| **호출** | User-Invoked (`/`로 직접) | Model-Invoked (키워드 자동) 또는 User-Invoked | Main이 위임 |
| **컨텍스트** | 메인 세션 안 | 메인 세션 안 | **격리된 새 세션** |
| **병렬 처리** | 불가 | 불가 (메인이 순차 실행) | **가능 (다수 동시)** |
| **장기 누적** | 자주 쓰는 명령 모음 | SOP·노하우 코드화 | 도메인별 전문 위임 |
| **변화 흐름** | 자주 쓰는 것 → Skill로 통합 | 반복 사용으로 고도화 | 좋은 레퍼런스 참조해 깎기 |

> 강사: "Slash command를 스킬로 통합했다 — 이 사실만 알고 있으면 됨" (옵시디언 실습.md 줄 167-169)

## Sub-Agent의 진짜 가치 (★ 핵심)

YouTube에서 광고하는 "마케팅 에이전트 팀을 만들어서 한 방에" 류는 **환상**.
실제 가치는 두 가지:

1. **격리된 컨텍스트** — 메인 대화 맥락에 영향 안 줌. 새 채팅을 백그라운드에서 만들어서 독립적으로 일함. 메인 토큰을 거의 안 먹지만 전체 토큰량에는 합산됨.
2. **병렬 처리** — 동일 스킬을 15명분에 동시 적용 가능. Sub-Agent에 스킬을 쥐여줄 수 있다.

> "에이전트는 스킬을 쥐고 사용할 수 있음" (옵시디언 260517.md 줄 259)
> "Main Agent = 대화창 자체 = 능력 있는 팀장. Sub-Agent = 한 영역을 가장 잘해오는 전문가." (`Q-Main-Sub-정의`)

## 커스텀 Sub-Agent 만들기 — 5단계

### 1. 막연하면 Thinking Partner부터
- `/thinking-partner`로 업무·니즈를 함께 정리
- 소크라테스식 질문으로 빈 공간을 채워줌

### 2. `/37.00-official-docs/` 폴더의 Sub-Agent 가이드 위치 확인
- `30-knowledge/37-claude-code/37.00-official-docs/` 안에 가이드 MD 파일이 있어야 함
- 없다면 `/doc-updater`로 공식 문서를 최신화

### 3. 가이드를 `@`-mention으로 제공
```
@30-knowledge/37-claude-code/37.00-official-docs/subagent-guide.md
이 가이드 기준으로 서브 에이전트를 만들어줘
```

### 4. 좋은 레퍼런스 추가 (★ 핵심)
한 줄 추가가 결과를 결정한다:
```
현재 Research Worker 정도의 깊이감이 있는 에이전트가 되면 좋겠어
```
→ 이렇게 하면 만들 때 **방법론을 많이 추가**해서 만들어진다. (`Q-레퍼런스-깊이감`)

### 5. 도메인 명시 + 의도 설명
```
내 글들을 동시에 여러 개 분석해서 AI스러운지 체크하는 에이전트가 필요해
```
- 슬래시 메뉴(`/agent` → Create New Agent)는 굳이 안 써도 됨
- 의도를 잘 설명하는 것만으로 충분 (`Q-에이전트-제작-법`)

## Skill 운영 팁

### 변화 흐름
- 초기 1회 만든 스킬 사용 → 새로운 케이스 발생 → 스킬 업데이트 → 반복
- "디테일한 회신 메일, 무시 메일, 그런것들에 대한 구체적 피드백 과정을 통해서 스킬자체가 고도화되는 과정이 중요함" (옵시디언 실습.md 줄 244)

### Escalation Pattern (★)
규칙의 강도를 사용자 피드백 누적으로 자동 승격:
- 1회 적발: **소프트 경고** ("이 강도는 아니야")
- 3회 누적: **"절대 금지"** 수준으로 자동 승격

> "처음에 한 번 걸린 거는 그 강도가 아니야 [...] 세 번 이상 걸려야 여기로 가는 거예요." (`Q-Escalation-3회`)

### Specialty를 뾰족하게 만들기
> 참석자: "사람처럼 일단 Specialty를 계속 뾰족하게 만드는 게 좋겠네요."
> 강사: "그게 닦는 거죠." (`Q-Specialty-닦기`)

## 체크리스트

- [ ] 이 작업이 **반복**되는가? (반복 없으면 일반 채팅으로 충분)
- [ ] **격리된 맥락**이 필요한가? (메인 맥락 오염 우려 시 Sub-Agent)
- [ ] **병렬 처리**가 필요한가? (다수 동시 처리 시 Sub-Agent)
- [ ] 좋은 레퍼런스를 같이 제공했는가? (Research Worker 같은)
- [ ] 출처·근거를 가져오도록 시스템 프롬프트에 명시했는가?

## 흔한 함정

- **"에이전트 팀" 환상**: 한 방에 다 되는 마법 없음. 오케스트레이션 개념으로 접근. (`Q-마케팅환상-비판`)
- **Sub-Agent 남발**: 격리·병렬이 필요 없으면 Skill로 충분. Sub-Agent는 토큰 비용이 든다.
- **레퍼런스 없이 만들기**: "그냥 만들어줘" → 깊이 없는 결과. 항상 좋은 레퍼런스를 같이 제공.
- **MCP 과다 설치**: 강력하지만 컨텍스트 윈도우를 점유. 필요한 것만. (옵시디언 실습.md 줄 209-212)
